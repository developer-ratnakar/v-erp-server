# Upcoming ERP Modules Architecture

This repo is still small enough that we can standardize new modules before the codebase spreads.

## Recommended module split

- `academic`
  - `programs`
  - `departments`
  - `batches`
  - `semesters`
  - `sessions`
- `students`
  - `students`
  - `parents`
  - `addresses`
  - `documents`
- `operations`
  - `subjects`
  - `timetables`
  - `attendance`
- `hr`
  - `staff`
- `exams`
  - `exams`
  - `results`
  - `marks`

Each module should follow the same internal shape already used in `auth` and `rbac`:

- `controllers/`
- `services/`
- `repositories/`
- `models/`
- `routes/`
- `validation/`
- optional `dto/`

## Standards to keep

- Keep API routes resource-oriented:
  - `/api/academic/programs`
  - `/api/students`
  - `/api/students/:studentId/addresses`
  - `/api/operations/subjects`
  - `/api/exams/:examId/marks`
- Put input validation in route middleware using `zod`.
- Keep controllers thin and move business rules into services.
- Keep repositories focused on Supabase/Postgres access only.
- Throw `ApiError` from services for user-facing failures.
- Use DB migrations as the source of truth for schema changes.

## Adjustments from the supplied feature spec

The supplied schema is close, but it needs a few changes to fit this repo cleanly.

### 1. Keep `users.id` as UUID

The current system already uses UUIDs in `public.users`. Because of that:

- `hr_staff.user_id` must be `UUID`, not integer.
- New domain tables can still use `BIGINT` identity primary keys.
- Cross-domain foreign keys should match the referenced table type exactly.

### 2. Use `BIGINT GENERATED ... AS IDENTITY` for business tables

For large operational tables, integer-style IDs are fine, but in Postgres the cleaner choice is:

- `BIGINT GENERATED BY DEFAULT AS IDENTITY`

This scales better than plain `SERIAL` and is the more modern pattern.

### 3. Use `TIMESTAMPTZ`

All timestamps should use timezone-aware columns:

- `created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`
- `updated_at TIMESTAMPTZ`

Use the existing `update_updated_at_column()` trigger helper.

### 4. Add explicit uniqueness and check constraints

Do not rely only on application logic for data quality.

Examples:

- unique `students.registration_number`
- unique `students.apaar_id`
- unique `operations_subjects.code`
- unique `academic_programs.code`
- unique `academic_departments.code`
- unique `academic_semesters.code`
- unique `academic_sessions.name`
- unique attendance on `(student_id, subject_id, month)`
- date-range checks such as `end_date >= start_date`

### 5. Prefer clear lookup/reference columns over derived duplication

Snapshot columns in attendance are acceptable for reporting, but they should be nullable and documented as denormalized filters only.

### 6. Keep file metadata in DB, binaries out of DB

Document tables should store:

- `file_url`
- optional `mime_type`
- optional `file_name`

Actual files should live in storage, not database blobs.

## Suggested implementation order

1. `academic`
2. `hr`
3. `students`
4. `operations` (`subjects`, `timetables`)
5. `attendance`
6. `exams`

That order minimizes foreign-key churn and lets downstream modules build on stable lookup data.
